# 🔀 PlanGo Frontend PR Rules (Pull Request 규칙)

PlanGo FE 레포에서는 협업 효율성과 코드 품질 유지를 위해 아래의 PR 규칙을 사용합니다.

---

# 1. PR 생성 대상 브랜치

- 모든 PR은 **dev 브랜치로 생성**한다.
- **main 브랜치로 직접 PR 금지** (main은 배포/완성본용)
- 기능 개발 → feat/* → dev  
- 버그 수정 → fix/* → dev  
- 디자인 수정 → design/* → dev  

---

# 2. PR 제목 규칙

형식:
<type>: <작업 내용 요약>

예시:
feat: 로그인 화면 UI 구현  
fix: 홈 카드 높이 오류 수정  
design: 버튼 라운드 스타일 변경  
docs: PR 규칙 문서 추가  

---

# 3. PR 내용(description) 작성 규칙

PR 본문에는 다음 정보를 반드시 포함한다.

## ✔ 필수 포함 항목
- 작업 내용 요약 (bullet 형식)
- 변경된 파일/구조 설명
- UI 변경 시 **스크린샷 필수**
- 테스트 방법 (어떻게 확인했는지)
- 체크리스트

예시 템플릿:
- 로그인 화면 UI 구성
- EditText 입력 스타일 적용
- 네비게이션 연결 완료

---

# 4. 스크린샷 규칙 (UI 변경 시 필수)

UI 관련 작업일 경우 PR에 반드시 스크린샷 첨부  
예:
- 로그인 화면 캡처
- 홈 카드 UI 변경 전/후 비교 이미지

---

# 5. PR 체크리스트

PR 본문에 아래 항목을 모두 포함한다.

- [ ] 정상적으로 빌드됨  
- [ ] Figma UI와 일치  
- [ ] 불필요한 코드 제거됨  
- [ ] 브랜치 규칙/커밋 규칙 준수  
- [ ] 충돌 여부 확인  

---

# 6. 리뷰 & 머지(Merge) 규칙

- 팀원 **1명 이상 승인(approve)** 후 merge 가능
- merge 방식은 **Squash & Merge** 사용  
  → 커밋 로그를 깔끔하게 유지하기 위함

---

# 7. PR 생성 예시

제목:
feat: 장소 검색 리스트 UI 구현

본문:
- RecyclerView로 장소 목록 구성  
- 아이템 카드 UI 추가  
- 네비게이션 연결  
- 검색창 디자인 적용  

스크린샷:
(이미지 첨부)

체크리스트:
- [x] 빌드 정상  
- [x] 디자인 일치  
- [x] 코드 리뷰 완료  

---

# ✔ 핵심 요약

- PR 대상은 dev  
- main 직접 PR 금지  
- UI 변경 시 스크린샷 필수  
- PR 제목은 <type>: <내용>  
- Squash & Merge 사용
